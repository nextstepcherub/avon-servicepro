1. EXECUTIVE SUMMARY
----------------------------------------------------------------------------------------------------------------
This migration report details the implementation of the Administration Module for Avon ServicePro Enterprise 
Database V1.0 in Supabase PostgreSQL. This module forms the foundational authentication, authorization, and 
profile layer of the entire system, using Supabase auth.users as its core authentication source.

A major challenge was the co-existence of two different data paradigms in the current codebase:
- Legacy V1 paradigm (Express Node.js API), which operates with local MySQL patterns, camelCase keys, flat VARCHAR 
  columns, and specific repositories (`user.repository.ts`, `rbac.repository.ts`).
- Modern V2 paradigm (Next.js & Supabase Frontend), which utilizes native PostgreSQL UUID keys, SNAKE_CASE, enums,
  arrays, triggers, and Row Level Security (RLS).

To resolve this seamlessly without breaking any existing repository operations or forcing a full API rewrite, 
we have designed a "Cross-Paradigm Synchronization Engine" using advanced PostgreSQL triggers. These triggers 
automatically map and synchronize columns between both paradigms in real-time, enabling the Node.js API and 
the Next.js client to query the same `user_profiles` table without conflict.

All tables, RLS policies, triggers, seed scripts, and verification scripts have been integrated into a single,
production-ready, atomic transaction SQL migration script located at:
`/src/db/sprint1_administration_module.sql`

3. DETAILED TABLE SCHEMAS (DATA DICTIONARY)
----------------------------------------------------------------------------------------------------------------

3.1 public.user_profiles
----------------------------------------------------------------------------------------------------------------
- id (UUID, Primary Key): References auth.users(id) ON DELETE CASCADE.
- email (VARCHAR(255), Unique, Not Null): User registration email.
- name (VARCHAR(150), Not Null): Used by V1 API for user's display name.
- full_name (VARCHAR(150), Not Null): Used by V2 Next.js frontend. (Synced automatically via trigger)
- role (VARCHAR(100), Default 'Technician'): Used by V1 API.
- role_v2 (public.avon_role, Default 'Technician'): Used by V2 frontend. (Synced automatically via trigger)
- tags (TEXT, Default '[]'): Used by V1 API as JSON string array.
- engineer_tags (public.engineer_tag[], Default '{}'): Used by V2 frontend. (Synced automatically via trigger)
- phone_number (VARCHAR(50)): Used by V2 frontend.
- department (VARCHAR(100), Default 'Field Technical Service'): Used by V2 frontend.
- avatarUrl (TEXT): Used by V1 API.
- avatar_url (TEXT): Used by V2 frontend. (Synced automatically via trigger)
- territory (VARCHAR(100)): For field engineer routing.
- passwordHash (VARCHAR(255)): Storage fallback for offline API testing.
- active / is_active (BOOLEAN, Default true): General active status indicators.
- created_at / updated_at (TIMESTAMPTZ): Audit control dates.

3.2 public.roles
----------------------------------------------------------------------------------------------------------------
- id (UUID, Primary Key, Default gen_random_uuid()): Unique identifier.
- name (VARCHAR(100), Unique, Not Null): e.g. 'Workshop Manager', 'Senior Biomedical Engineer'.
- description (TEXT): Detailed role duties.
- level (INTEGER, Default 1): Integer rank of authority (1-10) for authorization checks.
- active (BOOLEAN, Default true): Activation flag.
- created_at / updated_at / created_by / updated_by: Standard audit metadata.

3.3 public.permissions
----------------------------------------------------------------------------------------------------------------
- id (UUID, Primary Key, Default gen_random_uuid()): Unique identifier.
- code (VARCHAR(100), Unique, Not Null): e.g. 'jobs:create', 'reports:signoff'.
- description (TEXT): Scope of the privilege.
- active (BOOLEAN, Default true): Activation flag.

3.4 public.role_permissions
----------------------------------------------------------------------------------------------------------------
- id (UUID, Primary Key, Default gen_random_uuid()): Unique identifier.
- role_id (UUID, Foreign Key): References public.roles(id) ON DELETE CASCADE.
- permission_id (UUID, Foreign Key): References public.permissions(id) ON DELETE CASCADE.
- active (BOOLEAN, Default true): Mapping active flag.
- UNIQUE Constraint: role_id + permission_id (prevents duplicate mapping).

3.5 public.user_roles
----------------------------------------------------------------------------------------------------------------
- id (UUID, Primary Key, Default gen_random_uuid()): Unique identifier.
- user_id (UUID, Foreign Key): References public.user_profiles(id) ON DELETE CASCADE.
- role_id (UUID, Foreign Key): References public.roles(id) ON DELETE CASCADE.
- UNIQUE Constraint: user_id + role_id (prevents duplicate assignments).

3.6 public.engineer_tags
----------------------------------------------------------------------------------------------------------------
- id (UUID, Primary Key, Default gen_random_uuid()): Unique identifier.
- tag (VARCHAR(100), Unique, Not Null): e.g. 'Area Engineer', 'Workshop Engineer'.
- description (TEXT): Scope of tag specialization.

3.7 public.user_engineer_tags
----------------------------------------------------------------------------------------------------------------
- id (UUID, Primary Key, Default gen_random_uuid()): Unique identifier.
- user_id (UUID, Foreign Key): References public.user_profiles(id) ON DELETE CASCADE.
- tag_id (UUID, Foreign Key): References public.engineer_tags(id) ON DELETE CASCADE.
- UNIQUE Constraint: user_id + tag_id.


4. CROSS-PARADIGM SYNCHRONIZATION & AUTO-PROFILING ENGINE
----------------------------------------------------------------------------------------------------------------
To support both V1 and V2 codebase layers simultaneously, two specialized PostgreSQL triggers are implemented:

4.1 trg_sync_user_profiles
- Function: `public.sync_user_profiles_columns()`
- Frequency: BEFORE INSERT OR UPDATE ON public.user_profiles, per row.
- Operations:
  - If `name` is written by V1 repository, updates `full_name` for V2, and vice-versa.
  - If `avatarUrl` is written, updates `avatar_url` and vice-versa.
  - If `role` text is updated, parses and updates the `role_v2` enum column safely.
  - If `tags` JSON string is updated (e.g. '["Area Engineer"]'), parses the elements and updates the `engineer_tags` 
    enum array column (`{Area Engineer}`). If `engineer_tags` is written, exports to a serialized JSON string in `tags`.
  - Syncs `active` and `is_active` flags automatically.

4.2 on_auth_user_created
- Function: `public.handle_new_supabase_user()`
- Frequency: AFTER INSERT ON auth.users, per row.
- Operations:
  - Automatically captures user registration on Supabase Authentication.
  - Extracts raw user metadata (`full_name`, `role`, `department`) provided during signup.
  - Provisions a matching `public.user_profiles` record instantly with standard defaults, eliminating manual profile
    creation logic.


5. ROW LEVEL SECURITY (RLS) POLICIES
----------------------------------------------------------------------------------------------------------------
Supabase Row Level Security is enabled across all 7 core tables to enforce a multi-tenant-like permission boundary.

5.1 Helper Functions (Security Contexts)
- `public.get_auth_user_role()`: Retrieves the caller's active role enum securely.
- `public.is_workshop_manager()`: Returns TRUE if the current user is an active 'Workshop Manager'.
- `public.is_admin_officer()`: Returns TRUE if the current user is a 'Workshop Manager' or 'Documentation Officer'.

5.2 Configured Policies Matrix
- user_profiles:
  - SELECT: Authenticated users can view active profiles OR their own profile.
  - INSERT: Restricted to 'Workshop Manager'.
  - UPDATE: Allowed for profile owners OR 'Workshop Manager'.
  - DELETE: Restricted to 'Workshop Manager'.
- roles / permissions / role_permissions / engineer_tags:
  - SELECT: Allowed for all authenticated employees.
  - INSERT / UPDATE / DELETE: Restricted to 'Workshop Manager'.
- user_roles / user_engineer_tags:
  - SELECT: Allowed for all authenticated employees.
  - ALL (Management): Restricted to 'Workshop Manager'.


6. ENTERPRISE SEED DATA
----------------------------------------------------------------------------------------------------------------
To establish a fully functional RBAC system on boot, the following seed data is injected:

6.1 Role Definitions Seeding
- Complete set of 16 professional ranks (from Workshop Manager, Documentation Officer, down to Intern Technician).

6.2 Permissions Matrix Seeding
- 18 high-resolution privileges covering clinical instrument management, assets, customers, reporting 
  certificates (`reports:signoff`), and procurement authorizations (`procurement:approve`).

6.3 Role Permission Mappings
- Workshop Manager: Assigned ALL 18 permissions (Super admin).
- Documentation Officer: Assigned read permissions, update permissions, and certification signoffs.
- Biomedical Engineer: Assigned field operational permissions (`jobs:read`, `jobs:update`, `assets:read`).
- Technician: Assigned core field service progress updates.

6.4 Engineer Tags Seeding
- 'Area Engineer', 'Workshop Engineer', and 'Calibration Engineer' specializations.
